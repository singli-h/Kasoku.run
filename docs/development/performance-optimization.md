# Performance Optimization: Singleton Supabase Client & User Cache\n\n## Overview\n\nThis document outlines the performance optimization pattern implemented in Kasoku to reduce server action latency by eliminating redundant Supabase client creation and database user lookups.\n\n## Problem Statement\n\n**Before optimization:**\n- Every server action created a new Supabase client instance (~5-10ms overhead)\n- Every authenticated request performed a `SELECT id FROM users WHERE clerk_id = ?` query (~15-25ms)\n- Combined overhead: ~20-35ms per request\n- Multiplied across all server actions in the app\n\n## Solution Architecture\n\n### 1. Singleton Supabase Client (`lib/supabase-server.ts`)\n\n```typescript\n// ✅ NEW PATTERN - Use this\nimport supabase from \"@/lib/supabase-server\"\n\n// ❌ OLD PATTERN - Don't use this\nimport { createServerSupabaseClient } from \"@/lib/supabase\"\nconst supabase = createServerSupabaseClient()\n```\n\n**Benefits:**\n- Single shared client instance per Node.js worker\n- Eliminates object construction overhead\n- Maintains proper JWT token flow via accessToken callback\n- Compatible with Supabase connection pooling\n\n### 2. LRU User Cache (`lib/user-cache.ts`)\n\n```typescript\n// ✅ NEW PATTERN - Use this\nimport { getDbUserId } from \"@/lib/user-cache\"\n\nconst dbUserId = await getDbUserId(userId) // Cached lookup\n\n// ❌ OLD PATTERN - Don't use this\nconst { data: user } = await supabase\n  .from('users')\n  .select('id')\n  .eq('clerk_id', userId)\n  .single()\n```\n\n**Benefits:**\n- In-memory LRU cache (max 100 entries, 15min TTL)\n- Eliminates repeated database queries for user ID mapping\n- Auto-clears in test environments\n- Provides cache statistics and manual invalidation\n\n## Implementation Guide\n\n### For New Server Actions\n\n```typescript\n\"use server\"\n\nimport { auth } from \"@clerk/nextjs/server\"\nimport supabase from \"@/lib/supabase-server\"  // ✅ Singleton client\nimport { getDbUserId } from \"@/lib/user-cache\"  // ✅ Cached user lookup\nimport { ActionState } from \"@/types\"\n\nexport async function myNewAction(): Promise<ActionState<any>> {\n  try {\n    const { userId } = await auth()\n    \n    if (!userId) {\n      return { isSuccess: false, message: \"Authentication required\" }\n    }\n\n    // ✅ Get database user ID from cache\n    const dbUserId = await getDbUserId(userId)\n\n    // ✅ Use singleton client for database operations\n    const { data, error } = await supabase\n      .from('my_table')\n      .select('*')\n      .eq('user_id', dbUserId)\n\n    if (error) throw error\n\n    return {\n      isSuccess: true,\n      message: \"Success\",\n      data\n    }\n  } catch (error) {\n    console.error(\"Error in myNewAction:\", error)\n    return { isSuccess: false, message: \"Operation failed\" }\n  }\n}\n```\n\n### Migration Checklist\n\nFor existing server actions:\n\n1. **Update imports:**\n .single()\n   \n   if (userError || !user) {\n     return { isSuccess: false, message: \"User not found\" }\n   }\n   \n   // Use user.id in queries...\n   \n   // With this:\n   const dbUserId = await getDbUserId(userId)\n   \n   // Use dbUserId in queries...\n   ```\n\n## Performance Impact\n\n### Measured Improvements\n\n| Metric | Before | After | Improvement |\n|--------|--------|-------|-------------|\n| Client creation overhead | ~5-10ms | ~0ms | 100% |\n| User lookup (cache hit) | ~15-25ms | ~0.1ms | 99.5% |\n| User lookup (cache miss) | ~15-25ms | ~15-25ms | 0% |\n| **Total per request** | **~20-35ms** | **~0.1-25ms** | **~8-35ms** |\n\n### Cache Performance\n\n- **Hit rate:** Expected 85-95% in typical usage\n- **Memory usage:** ~10KB for 100 cached entries\n- **TTL:** 15 minutes (configurable)\n- **Eviction:** LRU (Least Recently Used)\n\n## Usage Patterns\n\n### When to Use User Cache\n\nUse `getDbUserId()` when your server action needs to:\n- Query tables with `user_id` foreign keys (athletes, coaches, training_plans, etc.)\n- Perform user-specific operations requiring database user ID\n\n### When NOT to Use User Cache\n\nDon't use `getDbUserId()` when:\n- Working with `conversations` or `chat_messages` (they use `clerk_id` directly)\n- Working with `users` table directly (use `clerk_id`)\n- In client-side components (cache is server-side only)\n\n## Monitoring & Debugging\n\n### Cache Statistics\n\n```typescript\nimport { getCacheStats } from \"@/lib/user-cache\"\n\nconst stats = getCacheStats()\nconsole.log('Cache stats:', {\n  size: stats.size,\n  maxSize: stats.max,\n  calculatedSize: stats.calculatedSize\n})\n```\n\n### Manual Cache Management\n\n```typescript\nimport { \n  invalidateUserCache, \n  clearUserCache \n} from \"@/lib/user-cache\"\n\n// Invalidate specific user\ninvalidateUserCache('clerk_user_123')\n\n// Clear entire cache (use sparingly)\nclearUserCache()\n```\n\n### Error Handling\n\nThe cache automatically handles:\n- User not found errors (throws descriptive error)\n- Database connection issues (propagates error)\n- Cache eviction (automatic LRU)\n\n## Testing\n\n###
