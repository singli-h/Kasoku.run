MVP PRD: Lean Athletes Management Page

1. Vision & Scope
Goal: Provide coaches a single, focused page to invite and manage athletes and groups with minimal UI, consistent with 2025/2026 design best practices.
Approach:
Vision Statement: Enable coaches to quickly invite/add athletes and manage group assignments with a clean, modular UI and robust server actions.
In-Scope:
- Single consolidated page with two sections: Roster & Group Directory
- Invite/Add-by-email flow that auto-detects existing users vs. Clerk invite
- Minimal roster columns: Name, Email, Group, Age, Sex, Events (tags)
- Bulk operations: Add to Group, Move to Group, Remove from Group (select all supported)
- Group CRUD (name only) with athlete count
- Coach-only access and authorization
- Athlete group history tracking for audit trail
- Secure permissions model (coaches cannot delete athlete records)
Out-of-Scope:
- KPI/analytics widgets, charts, performance metrics
- Messaging/export bulk actions
- Group metadata beyond name (e.g., capacity, tags)
- Frontend history viewing (backend tracking only for now)

2. Personas & User Stories
Persona Definitions:
- Coach: Manages groups and athletes; needs fast invite/assignment tools.
- Athlete: Receives invite and is assigned to groups; no direct use here.
User Stories:
- As a coach, I want to invite an athlete by email so that they can join my group.
- As a coach, I want to add an existing athlete to a group so that I don’t duplicate users.
- As a coach, I want to select multiple athletes and move them to another group so that I can reorganize quickly.
- As a coach, I want to create/rename/delete a group so that I can manage my roster structure.

3. Features & Logical Dependency Chain
Feature & Component List:
- Server actions: inviteOrAttachAthleteAction, bulkAssignAthletesAction, bulkMoveAthletesAction, bulkRemoveAthletesAction, getRosterWithGroupCountsAction
- Consolidated page UI: RosterSection, GroupDirectorySection, InviteInlineForm, GroupSelectorModal
- Reuse existing: createAthleteGroupAction, updateAthleteGroupAction, deleteAthleteGroupAction, getCoachAthleteGroupsAction, assignAthleteToGroupAction, removeAthleteFromGroupAction, getAthletesByGroupAction
Dependency Chain:
1) Server actions & validations
2) Group CRUD and counts
3) Roster table with selection & actions
4) Invite/Add inline form
5) Bulk operations wiring & tests
Atomic Scope:
- Each action independently testable; UI sections isolated.

4. Success Metrics & Release Criteria & Timeline
Key Metrics:
- Invite success rate > 95%
- Bulk operation error rate < 2%
- Roster load < 1.5s for 200 athletes
Release Checkpoints:
- All server actions validated with unit tests
- UI operates without analytics widgets
- Access control enforced for coaches only
MVP Exit Criteria:
- Invite/Add flow works end-to-end (existing user & new invite)
- Bulk add/move/remove working with confirmations
Timeline:
- Actions: 1 day; UI wiring: 1 day; Tests: 0.5 day

5. Architecture Overview & Figma Visualization
Tech & Decision Summary:
- Next.js App Router; Clerk auth; Supabase DB with RLS; Server actions pattern per repository
- Follow existing modules in apps/web/actions/training and auth
Database Schema Review:
- Use existing tables: users, athletes (events JSON), athlete_groups, athlete_group_histories
- No schema change required; compute age from users.birthdate
- RLS policies updated: athlete_groups now has INSERT/UPDATE/DELETE policies
- History tracking: athlete_group_histories table used for audit trail
- Security: Coaches cannot DELETE athlete records (only remove from groups)

7. Risks & Mitigations
Technical Risk: Clerk invite API behavior differences; Mitigation: wrap in try/catch with clear error mapping.
Scope Risk: Bulk operations complexity; Mitigation: batch in small chunks and report partial results.
Security Risk: Coach reassigning athletes from other coaches; Mitigation: validate coach ownership for all operations.

8. Error Handling & Best Practices
Centralized Error Handling: Server actions return ActionState with precise messages and codes.
Retries & Fallbacks: Simple retry around Clerk invites; partial success results for bulk.
Validation & Sanitization: Zod schemas for inputs; validate email; validate group ownership; ensure athlete existence.
Pitfalls to Avoid: No analytics; no direct Supabase in components; keep actions in actions/[domain].

9. Analytics & Performance Monitoring
Event Tracking: capture invites_sent, athletes_assigned, bulk_move success/failure counts.
Performance Metrics: measure roster load time and bulk action duration.
